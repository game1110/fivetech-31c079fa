投流回傳與歸因鏈路(環節一·流量端技術實現)

v1.0 · 2026-06-17 · 四環節中環節一(流量端)的後端深挖,與 C端業務流程(環節二·站點運營細部)對稱。全景與分層見 BC行業全景_總覽 / 站點業務流程_架構圖;文字版見〈整合全文〉§2。
本質是 Facebook Conversions API(CAPI)/ pixel 的伺服器端 postback —— iGaming 因瀏覽器端 pixel 被攔截/封殺,改走 server-to-server 才可靠、且能控制回傳內容(深度轉化)。

一、註冊回傳時序(能歸因:把點擊接到註冊)

sequenceDiagram autonumber participant U as 使用者瀏覽器 participant FE as 前端 participant API as API伺服器 participant DB as 資料庫 participant CS as ConversionService participant FB as 第三方API(Facebook) participant MQ as QueueService participant RT as RetryConversion rect rgb(243,232,255) note over U,DB: 1. 點擊落地與打點(種 click-id,尚無 UID) U->>FE: GET /landing?fbclid=FB12345&pixel_id=PIX7890 FE->>FE: 解析 URL (fbclid, pixel_id) + 設置 Cookie FE->>API: POST /save-tracking { fbclid, pixel_id, sessionID:S123 } API->>DB: INSERT tracking_data { sessionID, fbclid, pixel_id } DB-->>API: 儲存成功 end rect rgb(207,250,254) note over U,CS: 2. 轉化事件與身分回填(sessionID → UID) U->>FE: 提交註冊表單(僅帶 sessionID) FE->>API: POST /conversion-event { sessionID:S123, event:register } API->>DB: SELECT fbclid, pixel_id FROM tracking_data WHERE sessionID=S123 DB-->>API: fbclid=FB12345, pixel_id=PIX7890 API->>DB: UPDATE tracking_data SET UID=USER001 WHERE sessionID=S123 API->>CS: processConversion(UID, event:register, fbclid, pixel_id) end rect rgb(220,252,231) note over CS,RT: 3. Postback 回傳 + 失敗重試(可靠性) CS->>FB: POST /facebook_api { event:register, UID, fbclid, pixel_id } FB-->>CS: { status:success, message:"Conversion recorded" } CS->>DB: INSERT conversion_logs { UID, event, status:success } alt Postback 失敗 CS->>MQ: enqueue retry task MQ->>RT: consume retry task RT->>FB: POST /facebook_api (retry) FB-->>RT: { status:success } RT->>DB: UPDATE conversion_logs SET retry_count++, status=success end end
讀法:① 落地時還沒 UID,先用臨時 sessionID 把廣告身分(fbclid + pixel_id)暫存進 tracking_data;② 註冊請求只帶 sessionID,伺服器反查回廣告身分並回填 UID —— 完成「匿名點擊→具名玩家」的縫合(站內版的 斷裂點 1 最小解);③ 把轉化 postback 回 FB,讓投放算法 lookalike 放大,失敗走 QueueService → RetryConversion 保證「至少一次送達」(回傳越快放大越快,丟事件=好量白買)。

二、資料模型(兩張表)

表欄位角色
tracking_datasessionID(臨時鍵)、fbclid、pixel_id、UID(註冊後回填)點擊側打點 + 身分縫合
conversion_logsUID、event、value、status、retry_count回傳側帳本 + 重試狀態機
第一節只用到 event=register(boolean 式「轉化了沒」);第三節把 event 擴到 first_deposit/purchase,並啟用 value 欄(金額式價值信號)。

三、首存 / LTV 回傳時序(歸對價值:把回傳接到付費)

第一節只回傳到「註冊」—— 量便宜、含金量低,FB 只按註冊成本(CPL)優化,買進的常是不充值的薄量。要讓算法去找「會付費的人」,得把回傳事件源沿玩家漏斗往下游推。這是 斷裂點 2(投放端拿不到下游 LTV) 在工程上的形態。
sequenceDiagram autonumber participant PAY as 站點錢包(支付回調) participant GP as 遊戲供應商(op_code 事件流) participant CS as ConversionService participant DB as 資料庫 participant ML as pLTV 預測模型 participant FB as 第三方API(Facebook) rect rgb(254,243,199) note over PAY,FB: A. 首存回傳(真付費信號 → value 優化) PAY->>CS: webhook 首存成功 { UID:USER001, value:30, currency:USDT } CS->>DB: SELECT fbclid, pixel_id FROM tracking_data WHERE UID=USER001 DB-->>CS: fbclid=FB12345, pixel_id=PIX7890 CS->>FB: POST /facebook_api { event:first_deposit, value:30, currency:USDT, fbclid, pixel_id, event_id } FB-->>CS: { status:success } CS->>DB: INSERT conversion_logs { UID, event:first_deposit, value:30 } end rect rgb(220,252,231) note over GP,FB: B. 遊戲行為 → 預測 pLTV → 即時回傳(ROAS 優化) GP->>CS: 進遊前 3 分鐘行為事件 { op_code:OP777, ... } CS->>DB: 還原廣告身分:op_code → UID → fbclid note over CS,DB: 跨端 ID 映射;斷一段(無 op_code↔UID)就回不去廣告 CS->>ML: 預測 pLTV(前 3 分鐘行為) ML-->>CS: pLTV=120 USDT(判定優質) CS->>FB: POST /facebook_api { event:purchase, value:120(pLTV), fbclid, pixel_id, event_id } FB-->>CS: { status:success } CS->>DB: UPDATE conversion_logs SET predicted_value=120 end
兩個關鍵改動:(1) 回傳值從 boolean 升級成 value —— 帶上金額(首存額 / 預測 pLTV),FB 才能做 value optimization / ROAS 出價,這是「按 LTV 出價」在 API 層的具體形態;(2) 事件源從前端擴到錢包與遊戲端 —— conversion_logs 上游從一個變三個,其中遊戲端那條必須先有 UID ↔ op_code 映射才接得通,沒打通就只能回傳到首存為止。

時間矛盾(為何要預測模型):首存可能隔天、真 LTV 要 3-7 天才看得出,但「回傳越快放大越快」等不起 → 用進遊前 3 分鐘行為跑模型預測 pLTV,預測優質就即時回傳,把「準確但慢的真值」換成「即時但估的預測值」。這也是遊戲端子彈級事件數據餵投流端的最直接變現點。

四、回傳事件的深度階梯

回傳事件發生位置對 FB 的意義取得難度
register 註冊站點前端(第一節)激活,CPL 優化易,本站內就有
first_deposit 首存站點錢包(支付回調)真付費信號,CPA 優化中,需掛錢包 webhook
復充 / 累計付費站點錢包價值信號,可帶金額做 value 優化中
遊戲行為 / pLTV遊戲供應商端事件流ROAS / 按預期價值出價難,跨 op_code 體系(斷裂點 1)
越往下含金量越高、但發生越晚、跨的 ID 體系越多 —— 這就是「斷裂點 2 要靠斷裂點 1 先解」的根因。

五、陷阱點(合規 / 技術)

點說明對應全景
合規 落地頁斗篷(cloaking)/landing 對審核機器人與真人各回不同頁,是投 Meta 的日常手段;被抓即封 pixel/帳號§2 渠道分類「需斗篷、養號」
技術 sessionID 歸因窗口fbclid 有時效、Cookie 會掉、跨設備會斷 —— 註冊/首存隔太久反查不到,歸因漏掉算「自然量」斷裂點 1
技術 重試重複計數retry 必須對 FB 帶冪等鍵(event_id)去重,否則一個轉化回傳多次,虛增、誤導出價數據口徑(斷裂點 6)
技術 跨端 ID 映射遊戲行為要回到 fbclid,必須串起 sessionID → UID → op_code 三段;斷一段,遊戲端 LTV 就回不去廣告斷裂點 1
口徑 value 口徑回傳金額用首存額、毛 GGR 還是 NGR?口徑不一,FB 學到的「價值」就失真斷裂點 6
模型 預測偏差pLTV 預測偏高 → FB 拼命找「像高價值其實不充」的人,放大錯誤;模型要持續用真值回測校準—
合規 灰產回傳限制 / pixel 養號部分平台不收真實充值事件 → 只能回自定義「深度有效事件」當代理變量,本身就失真;pixel 被標記就燒掉,需可熱切換§2「封號是日常成本」
一句話定位:第一節是「能歸因」(把點擊接到註冊),第三節是「歸對價值」(把回傳接到付費與 LTV)。前者單站內就能做,後者必須跨站點錢包與遊戲端 —— 所以 斷裂點 2(投放端拿不到下游 LTV)本質上要靠斷裂點 1(ID 貫通)先解,兩者是同一條「數據跟著錢走」的鏈路在不同環節的斷口。
