集團 B2B2C 整合 — 初步發想

把分散的幾個 BU（流量端 / 站點 / 包網 / 遊戲供應商）當成一個產品來看，回答三個實際問題：每個 BU 怎麼連接、用戶旅程怎麼跑、運營要監控什麼。下面都是具體、可執行的想法。
⚠ 還在初步發想階段 · v0 · 待各 BU 訪談校正，非定論
為什麼這幾個 BU 能被串起來 —— 因為它們本來就共用兩條線
一條錢：全鏈只有一個錢源 = 玩家的淨輸（GGR）。投流、站點、包網、遊戲供應商的收入，全是同一筆 GGR 的切分與放大 —— 不是各賺各的。
一個人：從看到廣告到在遊戲裡輸贏，從頭到尾是同一個玩家。他不在乎背後是幾家公司拼的，他只經歷一條連續的旅程。
所以整合的本質很單純：讓「同一個玩家、同一筆錢、同一份數據」能從頭走到尾不斷棒。下面三節就是把這件事拆成 連接 → 旅程 → 監控。
① 每個 BU 怎麼連接 ② 用戶旅程怎麼跑 ③ 運營要監控什麼

① 每個 BU 怎麼連接 把鏈路畫出來：上游給下游什麼、下游付上游什麼、中間該交接什麼

BU 之間的「連接點」其實就是三次交接（handoff）。每一次交接都該同時遞三樣東西：玩家身分 + 金流口徑 + 行為數據。現在價值跟錢都遞了，獨獨「數據」這一棒一直掉 —— 這就是整合要補的縫。

客服 / 風控（橫向）跨層服務玩家—玩家遊戲內狀態給客服；黑名單 / 設備庫跨層共享各做各的
連接的關鍵一句話：價值跟錢的連接已經通了，缺的是數據這一棒。把每個交接點「該傳卻沒傳的數據」補上 —— 核心就是一個貫穿全鏈的玩家 ID，棒子才接得起來。這也是為什麼「先建數據中台 + 統一 ID」是其他所有事情的地基。

② 用戶旅程怎麼跑 同一個玩家，從曝光到流失，一條龍走過四個 BU

把玩家旅程從頭排到尾，標出每一步誰負責、用什麼系統、此刻他是什麼身分（ID）、現在追不追得到。會看到一件事：玩家是一條連續旅程，但他的身分換了三次，旅程被切成三段、沒人能從頭看到尾。
階段玩家在做什麼誰負責系統 / 觸點此刻的身分 ID現在追得到嗎
曝光滑到一則廣告流量端投放平台、素材渠道 ID投放側看得到
點擊 → 落地點進落地頁流量端落地頁（含斗篷）fbclid + sessionID站內暫存
註冊填表開帳號站點註冊/登入、設備指紋站點帳號 ID!渠道 ID → 站點 ID 第一次斷
首存第一次充值站點錢包 / 支付通道站點帳號 ID站點看得到，投流回不去
首投進遊戲下第一注遊戲供應商遊戲大廳 / 遊戲內op_code 體系!站點 ID → 遊戲 ID 第二次斷
活躍 / 復充反覆玩、反覆充站點 + 遊戲Promotion（紅利/反水）+ 遊戲內活動兩套並存站點促銷與遊戲內活動互盲
VIP 養成變成大 R站點VIP 體系、專屬客服站點帳號 ID看不到他在遊戲裡的狀態
沉睡一段時間不來站點CRM站點帳號 ID不知道他為什麼走
召回被訊息拉回站點SMS / Push / TG / 外呼站點帳號 ID只能猜內容
流失卸載 / 不回—重定向廣告已斷跨端拉回很難
旅程現在斷在哪：同一個玩家，從曝光到流失換了三套身分（渠道 ID → 站點帳號 → 遊戲 op_code），旅程被切成三截。結果是沒有人能從「他從哪個廣告來」一路看到「他在遊戲裡輸了多少」 —— 投流不知道買到的人值不值、站點看不到玩家在遊戲內卡哪、客服答不出玩家遊戲內的問題。
如果這條旅程能全程跟蹤，馬上能做四件事：① 投流按真實 LTV 出價（不再只按註冊成本買薄量）② 站點發的券能兌換到遊戲內（道具/免費場/專屬房），不再雙重讓利 ③ 客服看得到玩家遊戲內狀態，答得出「為什麼沒中獎」④ 流失預測知道玩家為何走、提前召回。這四件事全部依賴同一件前提：一個貫穿全鏈的玩家 ID（接回第①節）。

③ 運營要監控什麼 先盯一條龍漏斗，再盯各環節健康，最後盯異常

監控不是每個 BU 各看各的報表，而是順著「一條錢、一個玩家」往下盯。分三層：先看全鏈漏斗（玩家在哪一步漏掉、錢怎麼分），再看各環節自己的健康指標，最後是異常預警。
全鏈北極星（所有 BU 共同盯這兩個）
GGR（玩家淨輸 = 全鏈唯一錢源）　＋　玩家 LTV（一個玩家從進來到離開貢獻多少）
每個 BU 的成本與收益，都換算成「對 GGR / LTV 的貢獻」來看，才能共背、才不會各自為政。
第一層全鏈漏斗（跨 BU 一條龍）—— 最重要，現在最看不到
要盯的轉化看什麼漏在這裡代表
曝光 → 點擊 → 註冊各渠道 CPL、註冊成本素材/渠道問題（流量端）
註冊 → 首存註冊→首存轉化率、首存成本承接斷了（買到薄量 or 站點轉化差）
首存 → 首投 → 活躍前期留存（次留 / 7 留）、新手漏斗遊戲/新手體驗留不住（前期流失最兇）
活躍 → 復充 → VIP復充率、ARPPU、大 R 養成率變現/養客能力
全鏈 ROI從「買量花多少」到「最後 GGR 多少」的回本天數整條鏈賺不賺錢（現在算不出來）
現狀：這條漏斗橫跨四個 BU、四套 ID，現在拼不出來 —— 所以「最大卡點在哪」沒人答得準。第一層監控能跑，等於把 ID 貫通做完了（接回第①②節）。
第二層各環節健康（每個 BU 自己的儀表）
流量端CPL / 首存成本 CPA · 假量比例 · ROI 回本天數 · 各「渠道 × 市場」貢獻
站點（重點）註冊→首存轉化 · DAU · 次留/7留 · ARPPU · 紅利率 · 提款時效 · 代理貢獻占比
包網系統穩定/故障率 · 開站數 · 支付成功率 · 客戶續約率（B 端留存）
遊戲供應商實際 RTP vs 目標 · 各遊戲留存/生命週期 · 事件異常 · GGR 分成對帳
提款時效單獨拉出來盯 —— 它是玩家信任的生命線，提款慢/不到帳是這行最大的口碑殺手。
第三層異常預警（風控，跨層共享）
盯什麼異常抓什麼怎麼用
玩家行為異常升級/成長速度離譜、下注模式異常、套利、RTP 被拉高子彈級事件設「報警線」，超線即標記（遊戲端最有料）
流量異常某渠道假量、回傳灌水、註冊不充值的薄量清洗算法 + 回傳交叉驗證
資金異常充提異常、羊毛黨領完即走、提款異常充提風控規則
跨層黑名單同一套利團隊在三層分別出現共享黑名單 / 設備庫，一處抓到全鏈封
